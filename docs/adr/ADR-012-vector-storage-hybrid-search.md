# ADR-012: Unified Hybrid Search and Vector Storage via pgvector and Reciprocal Rank Fusion (RRF)

**Status:** Accepted  
**Date:** 2026-05-26  
**Decision Makers:** Lead Architect, High-Performance Implementer  

## Context

The Intelligent Second Brain (ISB.AI) requires storing document text chunks, metadata, and high-dimensional dense vector embeddings generated by our DroPE embedding model adapter. The system needs to support high-precision retrieval-augmented generation (RAG) that combines keyword-based search with semantic (dense vector) search. 

Key technical constraints and forces include:
1. **Self-Containment & Parity**: The architecture must support local workstation development (NVIDIA RTX 2060 GPU environment) and production cloud GKE deployment identically.
2. **Data Consistency**: Deletion requests (complying with LGPD owner-only purge cases, see [ADR-003](ADR-003-authorization-port.md)) must atomically delete the associated document metadata, text chunks, and vector embeddings.
3. **Isolation and CPU/GIL Limits**: Heavily concurrent semantic search calculations must not block our web server event-loop or cause GIL lock contention in the Python runtime.
4. **Relational Constraints**: Vector queries must support complex metadata filtering based on owner boundaries, date ranges, and category scopes.

## Decision

We will implement a unified vector storage and hybrid search architecture within our primary PostgreSQL database. We will enable the `pgvector` extension and configure Hierarchical Navigable Small World (HNSW) indexes for dense semantic vector similarity lookups. 

To achieve industrial-grade retrieval precision, we will combine sparse keyword-based search with dense vector search:
1. **Sparse Retrieval**: Leverages Postgres's native full-text search capability (`tsvector`, `tsquery`, and GIN indexes).
2. **Dense Retrieval**: Leverages `pgvector` cosine similarity (`<=>`) queries backed by an HNSW index.
3. **Score Integration**: Retrieve the top $N$ results from both sparse and dense queries and merge their ranking orders using Reciprocal Rank Fusion (RRF) in database-level queries or a clean SQL translator.
4. **Application Port**: We will encapsulate this capability behind a domain-agnostic `VectorStoragePort` defined in our application layer.

## Consequences

### Positive
* **ACID Transaction Integrity**: Document content, relational tags, and vector embeddings are modified atomically. Under LGPD purge actions, delete cascades remove associated vectors instantaneously, eliminating the risk of orphaned index entries.
* **Architecture Simplicity (KISS)**: No external managed vector databases (such as Pinecone) are added to the stack, keeping local docker-compose setup lightweight and preserving offline capabilities.
* **SQL Compatibility**: Full SQL syntax is supported. Relational joins, complex multi-tenant filters, and authorization constraints are handled natively by the database planner.
* **Performance Parity**: HNSW queries in Postgres run at single-digit millisecond scale for up to 5M vectors, meeting DORA-tier latency SLOs.

### Negative
* **Database Memory Constraints**: HNSW indexes are memory-bound. To maintain high query speeds, the index must reside in Postgres memory, requiring careful allocation of `shared_buffers` and `work_mem` as the database scales.
* **Index Build Overhead**: Building HNSW indexes is CPU and RAM intensive. We will run index builds asynchronously during database migration tasks or off-peak periods.

### Neutral
* **Indexing Algorithm Parameters**: Standard HNSW configuration parameters (`m` and `ef_construction`) are configured to prioritize recall over compilation speed, which is acceptable since indexes are built during deploy times.

## Alternatives Considered

### Option B: Distributed Dual-Store (Postgres + Pinecone Serverless)
* **Pros:** Auto-scales to hundreds of millions of vectors without Postgres resource contention.
* **Cons:** Introduces severe data consistency risks. We would need to manage reconciliation loops for synchronizing Postgres and Pinecone. It also violates strict multi-tenant privacy boundaries by shipping raw semantic metadata to a third-party cloud service, and breaks offline self-containment.
* **Why rejected:** Violates the KISS principle, breaks local workstation self-containment, and introduces eventual consistency synchronization lags.

### Option C: In-Process Vector Index (FAISS/LanceDB) inside Python Worker
* **Pros:** Zero database configuration; extremely fast memory queries.
* **Cons:** Runs inside the application worker memory. High-concurrency vector calculations block the Python GIL, freeze use-case loops, and lead to worker container OOMs. It also prevents clean horizontal scaling of worker containers since there is no unified state store.
* **Why rejected:** Violates container isolation and resource scaling constraints.

  - `PgVectorStorageAdapter` (infrastructure — pgvector and SQL full-text search adapter)

## Langfuse Ingestion Strategy

- **Trace Taxonomy**:
  - `trace_id`: Maps to the parent query execution or chat generation job ID.
  - Tags: `search_type` (hybrid/vector/text), `top_k`.
- **Span Hierarchy**:
  - Parent trace: `query` or `rag_pipeline`
  - Child spans:
    - `sparse_query` (tracks Postgres full-text search execution)
    - `dense_query` (tracks pgvector cosine distance execution)
    - `rrf_fusion` (tracks the reciprocal rank fusion sorting stage)
- **Prompt Version Tracking**: N/A (retrieval execution does not use dynamic prompts).
- **Score Schema**:
  - Tracks retrieval latency and recall@K quality metrics.
